# TabletopStoreUP — описание структуры проекта

Документ составлен по состоянию кодовой базы в каталоге `proj/`. Корень приложения Django: `TabletopStoreUP/`.

---

## 1. Общая схема

Проект — **интернет-магазин настольных игр** на **Django 5.2** с **Django REST Framework**, аутентификацией **JWT (SimpleJWT)** и БД **PostgreSQL**. Веб-интерфейс реализован шаблонами в приложении `store`; REST API объявлено в `TabletopStoreUP/urls.py`, логика — в `store/api.py` и `store/api_views.py`.

Развёртывание через **Docker Compose**: сервис `db` (PostgreSQL 15) и `web` (образ из `docker/Dockerfile`, Gunicorn).

---

## 2. Где хранятся данные о базе данных

### 2.1. Переменные окружения и файл `.env`

- **Файл:** `TabletopStoreUP/.env`  
- Загрузка: `python-dotenv` в `TabletopStoreUP/settings.py` — `load_dotenv(BASE_DIR / '.env', override=True)`.

Типичные переменные (имена; значения задаются локально и **не** коммитятся как секреты):

| Переменная   | Назначение |
|-------------|------------|
| `DB_ENGINE` | Движок Django, по умолчанию `django.db.backends.postgresql` |
| `DB_NAME`   | Имя базы данных |
| `DB_USER`   | Пользователь PostgreSQL |
| `DB_PASSWORD` | Пароль |
| `DB_HOST`   | Хост (`127.0.0.1` локально, в Docker — имя сервиса `db`) |
| `DB_PORT`   | Порт (обычно `5432`) |

### 2.2. Настройки Django

- **Файл:** `TabletopStoreUP/TabletopStoreUP/settings.py`  
- Словарь `DATABASES['default']` читается из переменных окружения через `os.getenv()`.

**Важно — имя БД по умолчанию:** в `settings.py` для `DB_NAME` fallback задан как `'TabletopStoreUp'` (буква **U** в конце в нижнем регистре), тогда как в `docker-compose.yml` и в документации чаще встречается **`TabletopStoreUP`**. Для согласованности лучше явно задавать `DB_NAME` в `.env` или в Compose.

### 2.3. Docker Compose

- **Файл:** `proj/docker-compose.yml`  
- Сервис `db`: образ `postgres:15`, переменные `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` подставляются из окружения с дефолтами:
  - `DB_NAME` по умолчанию: **`TabletopStoreUP`**
  - `DB_USER` по умолчанию: `postgres`
  - `DB_PASSWORD` по умолчанию: `postgres`

Сервис `web` получает `DB_HOST: db` и `DB_PORT: 5432`, чтобы Django подключался к контейнеру PostgreSQL по внутренней сети Compose.

### 2.4. Физическое хранение данных PostgreSQL в Docker

- Именованный том **`pgdata`** (секция `volumes` в `docker-compose.yml`) монтируется в контейнер БД как `/var/lib/postgresql/data`.  
- Данные кластера PostgreSQL живут в этом Docker-томе, а не в папке проекта на диске (если не настроено иное).

### 2.5. Резервные копии и медиа (файлы проекта)

- **`TabletopStoreUP/backups/`** — JSON-бэкапы (например, через management-команды / выгрузки).  
- **`TabletopStoreUP/media/`** — загружаемые файлы (изображения товаров и т.д.).  
- В `docker-compose.yml` для сервиса `web` смонтированы `./media`, `./backups`, `./staticfiles` в `/app/...` внутри контейнера.

## 3. Название базы данных

| Контекст | Значение по умолчанию |
|----------|------------------------|
| Docker Compose (`POSTGRES_DB` / `DB_NAME`) | **`TabletopStoreUP`** |
| `settings.py` (если `DB_NAME` не задан) | **`TabletopStoreUp`** (отличается регистром последней буквы) |

## 4. Где находится API (REST)

### 4.1. Точка входа URL

- **Файл:** `TabletopStoreUP/TabletopStoreUP/urls.py`

Префикс **`/api/`** подключает `DefaultRouter` DRF:

```text
path('api/', include(router.urls)),
```

Дополнительные маршруты аутентификации:

| Путь | Назначение |
|------|------------|
| `/api/auth/token/` | JWT: получение пары access/refresh |
| `/api/auth/token/refresh/` | Обновление access-токена |
| `/api/auth/token/verify/` | Проверка токена |
| `/api/auth/register/` | Регистрация (`RegisterView`) |

### 4.2. Реализация ViewSet’ов и сериализаторов

- **`TabletopStoreUP/store/api.py`** — основные ViewSet’ы: пользователи, товары, заказы, отзывы, справочники (роли, статусы, жанры, доставка, оплата и др.), `RegisterView`, дублирующийся фрагмент `MeUserSettingsViewSet` (см. ниже).  
- **`TabletopStoreUP/store/api_views.py`** — расширенный **`MeUserSettingsViewSet`** (настройки текущего пользователя, сохранение фильтров).  
- **`TabletopStoreUP/store/serializers.py`** — сериализаторы для API.  
- **`TabletopStoreUP/store/models.py`** — модели данных.

### 4.3. Регистрация маршрутов роутера (краткий перечень префиксов)

Под `/api/` (через router) доступны ресурсы вида:

- `user-roles/`, `order-statuses/`, `payment-statuses/`
- `delivery-methods/`, `delivery-statuses/`
- `genres/`, `player-ranges/`
- `users/`, `products/`, `orders/`, `order-items/`
- `payments/`, `deliveries/`, `profiles/`
- `reviews/`, `payment-methods/`
- `api/user/settings/` — настройки пользователя (вложенный префикс внутри `/api/` даёт путь вида `/api/api/user/settings/`)

Дополнительные действия у `ProductViewSet`: `products/top/`, `products/stats/`; у заказов: `orders/{id}/mark_paid/`.

### 4.4. Аутентификация API

Настройки в **`settings.py`**: `REST_FRAMEWORK` с `JWTAuthentication` и `SessionAuthentication`; `SIMPLE_JWT` — время жизни токенов и заголовок `Bearer`.

### 4.5. Мелкие HTTP-эндпоинты вне DRF-router

В **`store/urls.py`** (префикс приложения без `/api/` в корне проекта):

- `api/user/toggle-theme/` — переключение темы (для фронта на шаблонах).

Корзина и каталог в основном идут через **серверный рендеринг** (`views.py`), а не через перечисленный в README блок `/api/cart/` — при сверке с документацией имейте в виду расхождения README и фактических `urls.py`.

### 4.6. Примечание про `INSTALLED_APPS`

В `settings.py` указано приложение `'api'`. В дереве исходников проекта **отдельной папки приложения Django `api` нет**; в `requirements.txt` есть пакет **`api==0.0.7`** (PyPI). Это не то же самое, что модуль `store.api`. При ошибках загрузки приложения стоит проверить, задумано ли локальное приложение `api` или запись лишняя.

## 5. Docker: где что лежит и как подключается

| Файл | Роль |
|------|------|
| **`docker-compose.yml`** | Описание сервисов `db` и `web`, сеть, том `pgdata`, порты `5432` и `8000`, переменные окружения для веб-контейнера |
| **`docker/Dockerfile`** | Образ Python 3.12-slim, зависимости, `WORKDIR /app`, `PYTHONPATH=/app/TabletopStoreUP`, `DJANGO_SETTINGS_MODULE=TabletopStoreUP.settings`, `ENTRYPOINT` через `entrypoint.sh` |
| **`docker/entrypoint.sh`** | Ожидание доступности БД по TCP, `migrate`, `collectstatic`, опционально `seed_demo`, запуск **Gunicorn** на `0.0.0.0:8000` |
| **`.dockerignore`** | Исключает из контекста сборки `venv`, `.env`, медиа, бэкапы и т.д. (секреты из `.env` в образ не попадают — для контейнера задаются переменные в Compose) |

**Подключение Django к БД в Docker:** переменная **`DB_HOST=db`** (имя сервиса в Compose), порт **5432**. Скрипт в `entrypoint.sh` дополнительно ждёт открытия TCP до хоста `DB_HOST`.

**Запуск (из каталога `proj/`):**

```bash
docker compose up --build
```

Приложение будет доступно на порту **8000**, PostgreSQL при пробросе портов — на **5432** хоста (в `docker-compose.yml` порт БД открыт наружу; для продакшена часто убирают).

## 6. Структура каталогов проекта

Корень репозитория приложения: **`proj/`**.

```
proj/
├── docker-compose.yml          # Оркестрация: PostgreSQL + веб
├── docker/
│   ├── Dockerfile              # Сборка образа веб-приложения
│   └── entrypoint.sh           # Миграции, статика, Gunicorn
├── requirements.txt            # Python-зависимости
├── README.md                   # Установка, API, команды
├── .dockerignore
│
└── TabletopStoreUP/            # Рабочая директория Django (manage.py здесь)
    ├── manage.py
    ├── .env                    # Локальные секреты и DB_* (не для публикации)
    ├── locustfile.py           # Нагрузочное тестирование (Locust)
    │
    ├── TabletopStoreUP/        # Пакет настроек проекта
    │   ├── settings.py         # БД, DRF, JWT, почта, STATIC/MEDIA
    │   ├── urls.py             # Корневые URL: /api/, /admin/, store.urls
    │   ├── wsgi.py
    │   └── asgi.py
    │
    ├── store/                  # Основное приложение магазина
    │   ├── models.py
    │   ├── views.py            # Веб-представления (каталог, корзина, заказы)
    │   ├── urls.py             # Маршруты витрины (не префикс /api/)
    │   ├── api.py              # DRF ViewSet’ы для /api/
    │   ├── api_views.py        # Доп. API (настройки пользователя)
    │   ├── serializers.py
    │   ├── forms.py
    │   ├── admin.py
    │   ├── admin_reports.py    # Аналитика для админки
    │   ├── signals.py
    │   ├── permissions.py
    │   ├── migrations/         # История схемы БД
    │   ├── management/commands/  # backup, restore, seed_demo и др.
    │   ├── templates/          # HTML-шаблоны
    │   ├── static/             # Исходные статические файлы (часть в корне TabletopStoreUP/static)
    │   └── tests/
    │
    ├── static/                 # Статика разработки (css, js)
    ├── media/                  # Загрузки пользователей
    ├── backups/                # Файлы резервных копий
    └── staticfiles/            # Собранная статика (collectstatic; в Docker монтируется)
```

Виртуальное окружение **`venv/`** при локальной разработке обычно не включают в описание продакшена; в Docker оно не используется.

## 8. Полезные команды (локально)

Из каталога `TabletopStoreUP/` (где лежит `manage.py`):

```bash
python manage.py migrate
python manage.py runserver
python manage.py seed_demo
```

В контейнере те же команды вызываются из `entrypoint.sh` перед Gunicorn (кроме `runserver`).
